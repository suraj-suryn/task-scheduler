Project Overview

Task Scheduler is a production-ready, full-stack web application that combines a modern Spring Boot backend with a responsive Tailwind CSS UI. It implements three custom data structures (Interval Tree, Priority Heap, Dependency Graph) to handle real-world scheduling problems — overlap detection, priority ordering, and cycle prevention.

The application is deployed live on Railway with a Supabase PostgreSQL cloud database, featuring Spring Security role-based access control, a scheduled engine for auto-executing tasks, and a rich UI with charts, Kanban board, and dependency visualisation.

Data Structures & Algorithms

Three custom DSA classes are implemented in com.suraj.scheduler.dsa and hydrated from the database on application startup.

IntervalTree
Binary Search Tree

Purpose: Detects scheduling conflicts — prevents two tasks from having overlapping time intervals.


Key methods:
insert(Task)
isOverlapping(start, end, tasks)


Complexity: O(log n) insertion, O(log n) overlap query


Used in: TaskServiceImpl.saveTask()

PriorityHeap
Min/Max Heap

Purpose: Maintains tasks ordered by priority (1–10) for the scheduler engine and smart suggest feature.


Key methods:
add(Task)
poll()
peek()


Complexity: O(log n) insert/poll


Used in: TaskSchedulerEngine, /api/tasks/suggest

DependencyGraph
DAG + DFS Cycle Detection

Purpose: Stores task dependencies as a directed graph and detects cycles to prevent invalid dependency chains.


Key methods:
hasCycle(Task, List)
topologicalSort()


Complexity: O(V + E) DFS traversal


Used in: TaskServiceImpl.saveTask(), /tasks/graph

Key Configuration Notes

Why Session Pooler? Railway is IPv6-only. Supabase's direct connection uses IPv4 only. The Session Pooler endpoint supports IPv6 and uses port 5432 (not 6543 — that's the Transaction Pooler, which is incompatible with Hibernate DDL operations).

Why separate credentials? The DB password contains @ which breaks JDBC URL encoding if embedded inline. Separate DB_USERNAME and DB_PASSWORD env vars avoid this.

JPQL CAST fix: PostgreSQL couldn't infer the type of :search parameter in LOWER(), throwing a lower(bytea) error. Fixed by wrapping: LOWER(CONCAT('%', CAST(:search AS string), '%')).
