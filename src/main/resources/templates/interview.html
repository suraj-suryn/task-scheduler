Interview Q&A

Technical questions & answers about this Task Scheduler project

It's a personal task scheduling and tracking web application. The core problem: most to-do apps don't enforce time conflicts (you can schedule two tasks at the same time), don't validate task dependencies (task B starts before task A which it depends on), and don't automatically transition task statuses. This app solves all three with custom DSA — an IntervalTree for overlap detection, a DependencyGraph for cycle validation, and a scheduler engine for automatic status transitions.
Spring Boot 3.5 (Java 17) — production-ready, convention-over-configuration, built-in security and scheduling. Thymeleaf for server-side rendering — the app is CRUD-heavy with form submissions, so SSR gives faster initial load and simpler state management without a separate SPA. PostgreSQL on Supabase for persistence. Tailwind CSS CDN for responsive UI without a build pipeline. Deployed on Railway with auto-deploy on git push.
Standard 3-tier: Controller → Service → Repository. I kept two controllers separate — TaskController handles Thymeleaf views and TaskRestController handles JSON REST calls (status updates, smart suggest) — keeping view and API concerns cleanly separated. Business logic lives entirely in TaskServiceImpl. JPA repositories handle persistence. Custom DSA classes (IntervalTree, DependencyGraph, PriorityHeap) are used by the service layer.
Three custom DSA classes:

1. IntervalTree — detects time-overlapping tasks. Each node stores a [startTime, endTime] interval and the max endTime of its subtree. Overlap check is O(log n) vs O(n²) brute force.

2. PriorityHeap — min-heap ordering tasks by priority + deadline for the "What's Next?" suggestion. Insert O(log n), peek O(1).

3. DependencyGraph — adjacency list + DFS-based cycle detection to validate task dependency chains before save.
Each node stores its interval [start, end] and the max endTime of its entire subtree — this is what makes range queries efficient. When inserting a new task, we traverse the tree: if the current node's max is less than the new task's start, we can prune the entire subtree (no overlaps possible). Overlap condition: start1 < end2 && start2 < end1. If overlap found, a TaskOverlapException is thrown before the task is saved.
DependencyGraph builds a directed adjacency list where an edge from A → B means "A depends on B". Before saving any task, I run DFS with two sets: visited and recursionStack. If DFS encounters a node already in recursionStack, a cycle exists → throws DependencyCycleException. This prevents infinite loops in the scheduling engine.
TaskSchedulerEngine is a @Service using Spring's @Scheduled. It runs periodically and:

1. Queries tasks where startTime <= now AND status = PENDING → transitions to RUNNING
2. Queries tasks where endTime <= now AND status = RUNNING → transitions to COMPLETED
3. Overdue checks: tasks still PENDING past their endTime → FAILED
Two @ControllerAdvice classes: RestExceptionHandler for REST endpoints (returns JSON with HTTP status) and UiExceptionHandler for Thymeleaf views (sets errorMessage flash attribute and redirects). Custom exceptions: TaskOverlapException (409), DependencyCycleException (400), InvalidTaskTimeException (400), TaskNotFoundException (404). This keeps controllers clean — they never catch exceptions directly.
Separation of concerns. TaskController handles Thymeleaf form submissions and view rendering — it returns view names and uses redirect attributes. TaskRestController handles async JavaScript calls — inline status dropdown changes, the "What's Next?" smart suggest, CSV export — returning JSON or binary. Mixing both in one controller would create confusing method signatures and inconsistent error handling.
Task has: name, description, priority (int 1–10), startTime / endTime (LocalDateTime), status (enum: PENDING/RUNNING/COMPLETED/FAILED), recurrenceType (enum: NONE/DAILY/WEEKLY/MONTHLY), dependencies (String — comma-separated task IDs), category (@ManyToOne → Category), user (@ManyToOne → User). Relationships use lazy loading; category and user are joined on task list queries.
All service queries filter by the authenticated user: taskRepository.findByUser(currentUser). The current user is resolved from SecurityContextHolder via a SecurityUtils helper. Admins can see all tasks via a separate query path. Task IDs in URLs (delete, update) are verified to belong to the requesting user before any mutation — otherwise 403/404 is returned.
Spring Security with UserDetailsService + DaoAuthenticationProvider. Passwords hashed with BCrypt. Session-based authentication (Spring's default). Roles: ROLE_USER and ROLE_ADMIN. Email verification flow: on registration, a UUID token is stored on the User entity and emailed; the user must click the link before they can log in. CSRF protection is enabled for all form submissions (token injected via Thymeleaf's th:name="${_csrf.parameterName}").
A1 Broken Access Control — user-scoped queries, admin-only routes protected via .requestMatchers("/admin/**").hasAuthority("ROLE_ADMIN").
A2 Cryptographic Failures — BCrypt for passwords, never stored in plaintext.
A3 Injection — JPA repositories use parameterized queries, no raw SQL.
A7 CSRF — Spring Security CSRF tokens on all state-changing forms.
A5 Security Misconfiguration — credentials in environment variables, not hardcoded; H2 console restricted.
Sliding drawer sidebar — hidden off-screen with -translate-x-full on mobile, slides in with CSS transition on hamburger click. Mobile top bar with hamburger, logo, dark mode toggle, context-aware + button. Bottom navigation bar for quick page access. Tables wrapped in overflow-x-auto with min-width so they scroll horizontally. Tailwind breakpoints (sm:, lg:) control grid columns — e.g. stat cards go from 2-col mobile to 6-col desktop.
A Thymeleaf double-render bug. The th:fragment="layout(...)" was placed on a <th:block> inside an outer <html><body> wrapper, and the fragment itself contained another <!DOCTYPE html><html>. Thymeleaf's Attoparser broke on the nested <html> inside <body>, outputting </th:block> as literal text and rendering all content twice. The main flex wrapper got width: 0px breaking the entire mobile layout. Fix: move th:fragment directly onto the <html> element — the standard Thymeleaf pattern.
Railway with auto-deploy on git push to master. Railway detects the Maven wrapper, runs mvnw package, and starts the JAR. Database is PostgreSQL on Supabase. Spring profile prod activates the PostgreSQL datasource. The Procfile / Railway start command runs the packaged JAR. Zero-downtime deploys — Railway spins up the new instance before terminating the old one.
All secrets are in Railway environment variables — never hardcoded in source. application.properties reads them via ${DATABASE_URL}, ${DB_USERNAME}, ${DB_PASSWORD} placeholders. The git repository contains no credentials. Local development uses a separate application-local.properties that is gitignored.
